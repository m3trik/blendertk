# blendertk ↔ mayatk structure map

**Why this file exists:** `blendertk` mirrors `mayatk` so a change in one can be mirrored to the
other mechanically. Keep the two **structurally parallel** — same subpackage names, same main-
module filenames, same namespace class names — so "where does this live in the other package?"
is always answerable without searching. When you add/move/rename something in one package, update
the counterpart **and** this table.

This is the structural SSoT. For the *measured* parity gap (per-panel/menu/helper) see
[`tentacle/docs/PARITY_AUDIT.md`](../../tentacle/docs/PARITY_AUDIT.md) (auto-generated); for the
actionable port-this-next recipes see
[`tentacle/docs/PARITY_PORTING_PLAN.md`](../../tentacle/docs/PARITY_PORTING_PLAN.md); for the
public API surface see [`API_REGISTRY.md`](../API_REGISTRY.md). (The old `BLENDER_FEATURE_GAPS.md` /
`BLENDER_PORT_PLAN.md` were retired — those two audit docs supersede them.)

## Subpackage correspondence

Each row is `mayatk/mayatk/<sub>/` ↔ `blendertk/blendertk/<sub>/`. The **main module** is
`_<sub>.py` in both, exposing the **namespace class** (`mtk.<Class>` ↔ `btk.<Class>`).

**Rule for the ⛔ rows:** do **not** pre-create an empty `*_utils` package to match mayatk.
Create the subpackage only when a Blender slot needs a real engine helper there — at which point
use the **same subpackage + main-module name** as mayatk so the row flips to ✅.

## Conventions (identical in both packages)

- **Main module** per subpackage: `_<sub>.py` (e.g. `edit_utils/_edit_utils.py`), holding module-
  level functions **and** a `<Domain>Utils` namespace class whose members are `staticmethod`
  wrappers of those functions. Registered via the root `__init__.py` `DEFAULT_INCLUDE`.
- **Subpackage `__init__.py`** = docstring only (no import side effects).
- **`import bpy` / `import maya.cmds`** deferred into call bodies (resolving the package surface
  must never need a running DCC).
- **Co-located tool panels** live next to their engine in the relevant subpackage as `<tool>.py`
  (engine + `<Tool>Slots`) + `<tool>.ui` (the tracked Designer SSoT). The `<Tool>Slots` class is
  **discovered by the UI handler**, *not* listed in `DEFAULT_INCLUDE` (only the engine
  helpers/classes are). Same split in both packages.
  - **`<tool>_ui.py` is a generated, gitignored build artifact in both** (`*_ui.py` is in both
    `.gitignore`s) — mayatk compiles it with pyside6-uic; blendertk's tentacle runtime loader
    generates it from the `.ui`. Only the generation mechanism differs; the committed *source*
    (`<tool>.py` + `<tool>.ui`) is identical in layout across both.
  - **Tool size:** where mayatk splits a tool into a `<tool>/` **subpackage** (e.g.
    `uv_utils/rizom_bridge/` = `__init__.py` + `_rizom_bridge.py` engine + `rizom_bridge_slots.py` +
    `rizom_bridge.ui`), blendertk **mirrors that subpackage layout** (full 1:1 file mirror, decided
    2026-06-14 — see [`PARITY_BACKLOG.md`](PARITY_BACKLOG.md)). The implementation body stays
    Blender-appropriate (thin where mayatk's heavy machinery has little Blender benefit), but the
    file tree + class names match so a change maps mechanically.

## Registration style

Both call `bootstrap_package(globals(), include=DEFAULT_INCLUDE)`. mayatk's keys are mostly the
**legacy bare module name** (`"_edit_utils": "*"`) with wildcard export; blendertk uses the
**explicit full path + class list** (`"edit_utils._edit_utils": ["EditUtils", "decimate", …]`).
Both resolve identically through `pythontk.module_resolver` (a module absent from `include` is not
exposed). Prefer the explicit blendertk form for new entries — it's the controlled surface.

## Porting checklist (mirror a change from one package to the other)

1. Same **subpackage** + **main-module filename**? (If the target subpackage is a ⛔ row, create
   it with mayatk's name.)
2. Same **class name** + **public method/function name** (mirror *names + behavior*, not
   signatures — `cmds` string-nodes vs `bpy` object refs differ).
3. New public symbol → add to `DEFAULT_INCLUDE` (and verify `btk.<name>` / `mtk.<name>` resolves).
4. New co-located tool → `<tool>.py` + `<tool>.ui`; `<Tool>Slots` discovered by the handler.
5. Update this table and the package `CHANGELOG.md`; refresh `API_REGISTRY.md` if the surface
   changed (`python m3trik/scripts/generate_api_registry.py blendertk`).
